# 테스트와 반복 개선

스킬 테스트는 필요 수준에 따라 엄격도를 다르게 가져갈 수 있습니다.
- Claude.ai 수동 테스트: 질의를 직접 실행하고 동작을 관찰. 설정 없이 빠른 반복 가능.
- Claude Code 스크립트 테스트: 반복 가능한 검증을 위해 테스트 케이스 자동화.
- Skills API 프로그래밍 테스트: 정의된 테스트 세트를 체계적으로 평가하는 평가군 구축.

스킬의 품질 요구사항과 노출 범위에 맞는 방식을 선택하세요. 소규모 내부 팀에서만 쓰는 스킬과, 수천 명의 기업 사용자가 쓰는 스킬은 테스트 요구가 다릅니다.

## 실전 팁: 범위를 넓히기 전에 단일 과업에서 반복

가장 효과적인 제작자는 보통 난이도 있는 단일 과업에서 클로드가 성공할 때까지 반복하고, 그 성공 패턴을 스킬로 추출합니다. 이 방식은 클로드의 문맥 내 학습을 활용하며, 광범위 테스트보다 빠르게 신호를 얻을 수 있습니다. 동작 기반이 잡히면 그다음 여러 테스트 케이스로 범위를 확장하세요.

## 권장 테스트 접근

초기 경험 기준으로, 효과적인 스킬 테스트는 보통 아래 3가지를 포함합니다.

### 1. 트리거 테스트
목표: 스킬이 올바른 시점에 로드되는지 확인

#### 테스트 케이스
- ✅ 명백한 작업에서 트리거됨
- ✅ 같은 뜻의 바꿔 말한 요청에서도 트리거됨
- ❌ 무관한 주제에서는 트리거되지 않음

#### 예시 테스트 묶음

트리거되어야 함:
- "새 ProjectHub 작업공간 설정 도와줘"
- "ProjectHub에 프로젝트를 만들어야 해"
- "Q4 계획용 ProjectHub 프로젝트 초기화해줘"

트리거되면 안 됨:
- "샌프란시스코 날씨 어때?"
- "파이썬 코드 작성 도와줘"
- "스프레드시트 만들어줘"(ProjectHub 스킬이 시트를 다루지 않는다면)

### 2. 기능 테스트
목표: 스킬이 올바른 출력을 만드는지 검증

#### 테스트 케이스
- 유효한 출력 생성
- API 호출 성공
- 오류 처리 동작
- 경계 사례 커버

#### 예시

테스트: 작업 5개가 있는 프로젝트 생성
Given: 프로젝트 이름 `Q4 Planning`, 작업 설명 5개
When: 스킬이 워크플로 실행
Then:
- ProjectHub에 프로젝트 생성
- 올바른 속성으로 작업 5개 생성
- 모든 작업이 프로젝트에 연결
- API 오류 없음

### 3. 성능 비교
목표: 기준선 대비 스킬의 개선 효과 입증

`성공 기준 정의` 절의 지표를 사용하세요. 아래는 비교 예시입니다.

#### 기준선 비교

스킬 없이:
- 사용자가 매번 지침을 직접 입력
- 왕복 메시지 15회
- 재시도 필요한 API 실패 3회
- 토큰 12,000 사용

스킬 사용:
- 워크플로 자동 실행
- 확인 질문 2회만 발생
- API 실패 0회
- 토큰 6,000 사용

## `skill-creator` 스킬 활용

`skill-creator` 스킬은 Claude.ai(플러그인 디렉터리) 또는 Claude Code 다운로드로 사용할 수 있으며, 스킬 생성과 반복 개선에 도움을 줍니다. MCP 서버가 있고 핵심 워크플로 2~3개를 알고 있다면, 보통 한 번의 세션(대개 15~30분) 안에 기능하는 스킬을 만들고 테스트할 수 있습니다.

### 스킬 생성 지원
- 자연어 설명에서 스킬 초안 생성
- 프런트매터가 올바른 `SKILL.md` 생성
- 트리거 문구와 구조 제안

### 스킬 리뷰 지원
- 흔한 문제 탐지(모호한 설명, 트리거 누락, 구조 문제)
- 과도/부족 트리거 위험 식별
- 스킬 목적 기반 테스트 케이스 제안

### 반복 개선 지원
- 실제 사용 중 발견한 경계 사례/실패 사례를 `skill-creator`에 다시 입력
- 예시: "이 대화에서 확인한 문제와 해결책을 반영해서 [특정 경계 사례] 처리 방식을 개선해줘"

### 사용 방법

`[사용 사례]용 스킬을 만들 수 있게 skill-creator 스킬을 사용해줘`

참고: `skill-creator`는 설계·정제에는 유용하지만, 자동 테스트군 실행이나 정량 평가 결과 생성까지 대신하진 않습니다.

## 피드백 기반 반복

스킬은 살아있는 문서입니다. 아래 신호를 기준으로 반복 개선을 계획하세요.

### 부족 트리거 신호
- 트리거되어야 할 때 로드되지 않음
- 사용자가 수동으로 켜야 함
- "언제 써야 하나요?" 지원 문의 증가

해결: `description`에 더 구체적인 맥락과 뉘앙스를 추가(특히 기술 용어 키워드 보강)

### 과다 트리거 신호
- 무관한 질의에서 로드됨
- 사용자가 스킬을 끔
- 스킬 목적에 대한 혼란 발생

해결: 부정 트리거 추가, 범위를 더 구체화

### 실행 이슈
- 결과 일관성 부족
- API 호출 실패
- 사용자 수정 필요

해결: 지침 보강, 오류 처리 추가
