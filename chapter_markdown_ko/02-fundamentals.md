# 기초

## 스킬이란 무엇인가?

스킬은 다음을 포함하는 폴더입니다.
- `SKILL.md` (필수): YAML 프런트매터가 포함된 마크다운 지침
- `scripts/` (선택): 실행 가능한 코드(Python, Bash 등)
- `references/` (선택): 필요 시 불러오는 문서
- `assets/` (선택): 결과물에 쓰는 템플릿, 글꼴, 아이콘

## 핵심 설계 원칙

### 점진적 공개

스킬은 3단계 구조를 사용합니다.
- 1단계(YAML 프런트매터): 항상 클로드 시스템 프롬프트에 로드됩니다. 모든 내용을 문맥에 넣지 않고도, 어떤 상황에서 스킬을 써야 하는지 판단할 최소 정보를 제공합니다.
- 2단계(`SKILL.md` 본문): 클로드가 현재 작업에 스킬이 관련 있다고 판단할 때 로드됩니다. 전체 지침과 가이드를 담습니다.
- 3단계(연결 파일): 스킬 디렉터리에 포함된 추가 파일로, 필요할 때만 클로드가 탐색하고 불러옵니다.

이 점진적 공개 방식은 전문성을 유지하면서 토큰 사용량을 줄여줍니다.

### 조합 가능성

클로드는 여러 스킬을 동시에 로드할 수 있습니다. 따라서 내 스킬은 단독 사용만 가정하지 말고, 다른 스킬과 함께 동작하도록 설계해야 합니다.

### 이식성

스킬은 Claude.ai, Claude Code, API에서 동일하게 동작합니다. 필요한 의존성이 환경에 있으면, 한 번 만든 스킬을 수정 없이 여러 표면에서 재사용할 수 있습니다.

## MCP 구축자를 위한 관점: 스킬 + 커넥터

💡 MCP 없이 단독 스킬만 만든다면 이 절은 건너뛰고 `Planning and design`으로 가도 됩니다.

이미 동작하는 MCP 서버가 있다면 어려운 단계는 끝난 것입니다. 스킬은 그 위에 올라가는 지식 계층으로, 이미 알고 있는 워크플로와 모범 사례를 캡슐화해 클로드가 일관되게 적용하게 만듭니다.

### 주방 비유

MCP는 전문 주방입니다. 즉, 도구·재료·장비에 대한 접근을 제공합니다.
스킬은 레시피입니다. 즉, 가치 있는 결과를 만들기 위한 단계별 지침을 제공합니다.

둘을 함께 쓰면 사용자는 모든 단계를 직접 파악하지 않아도 복잡한 작업을 수행할 수 있습니다.

### 함께 동작하는 방식

MCP(연결성) / 스킬(지식)
- 클로드를 사용자의 서비스(Notion, Asana, Linear 등)에 연결함
- 클로드가 서비스를 효과적으로 쓰는 방법을 가르침
- 실시간 데이터 접근과 도구 호출을 제공함
- 워크플로와 모범 사례를 담아냄
- “클로드가 무엇을 할 수 있는가”를 제공함
- “클로드가 어떻게 해야 하는가”를 제공함

### 왜 MCP 사용자에게 중요한가

#### 스킬이 없으면
- 사용자는 MCP를 연결해도 다음에 무엇을 해야 할지 모름
- “이 통합으로 X를 어떻게 하나요?” 같은 지원 문의가 늘어남
- 매 대화가 처음부터 다시 시작됨
- 사용자마다 프롬프트가 달라 결과가 들쭉날쭉함
- 실제 문제는 워크플로 안내 부족인데 커넥터 문제로 오해받음

#### 스킬이 있으면
- 사전 구축된 워크플로가 필요할 때 자동으로 활성화됨
- 도구 사용이 일관되고 신뢰 가능해짐
- 모든 상호작용에 모범 사례가 기본 탑재됨
- 통합 학습 곡선이 낮아짐
